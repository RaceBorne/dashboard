# Evari dashboard audit — 2026-04-21

A deep pass over the repo looking for dead code, duplication, oversized files,
and recurring patterns that are slowing development down. Findings are
ordered by impact-to-effort: do the things at the top first.

## TL;DR — do these three things first

1. **Split the four giant component files.** `WireframeDiagram.tsx` (3,104),
   `TrafficDashboard.tsx` (1,523), `SeoHealthClient.tsx` (1,376),
   `PlayDetailClient.tsx` (1,334). Every edit to these now means scrolling
   for thirty seconds and fighting the AI's context window. One afternoon of
   splitting buys weeks of speed.
2. **Extract a shared `EditableString` / `EditableList` / `EditableNumber`
   component library** out of the inline-edit work just landed on
   `PlayDetailClient.tsx`. Every future detail view (drafts, senders,
   suppressions, tasks) will want the same click-to-edit pattern. Ship it
   once, reuse it five times.
3. **Build the `jsonb-route-scaffolder` subagent.** `PATCH /api/*/[id]` is
   written the same way six times now: validate each field -> trim -> merge ->
   log activity -> update `payload` jsonb. A subagent that takes a type name
   and a field list and spits out the route would have saved hours on the
   last two weeks.

Everything else is gravy.

## 2. Duplicated logic — highest ROI extractions

**`createSupabaseAdmin()` boilerplate.** Twenty-three distinct API route
files open with the same six lines:

```ts
const supabase = createSupabaseAdmin();
if (\!supabase) {
  return NextResponse.json(
    { ok: false, error: 'Supabase admin client unavailable' },
    { status: 500 },
  );
}
```

Extract a wrapper in `lib/supabase/api.ts`:

```ts
export function withAdmin<T>(
  handler: (supabase: SupabaseClient) => Promise<NextResponse<T>>,
) { ... }
```

Every PATCH/GET/DELETE drops one-sixth shorter. Cheap, safe, read-at-a-glance.

**Inline editors — see #2 in the TL;DR.** The three `<InlineText>`,
`<InlineList>`, `<InlineNumber>` components currently live inside
`PlayDetailClient.tsx`. Hoist to `components/ui/editable/` with no API
change and every future detail pane gets click-to-edit for free.

**PlayStrategy validation + merge.** `app/api/plays/[id]/route.ts` has a
`validateStrategyPatch` + `mergeStrategy` pair that is a clear template for
every future "partial update a jsonb sub-document" route. Generalise into
`lib/dashboard/patch.ts`:

```ts
export function validatePatch<T>(input: unknown, schema: FieldSchema<T>): Result<T>;
export function mergePatch<T>(existing: T | undefined, patch: Partial<T>): T;
```

and the new routes pick it up for free.

**Integration status cards.** `getShopifyStatus`, `getDataForSeoStatus`,
`getGoogleStatus` all follow the same cache-then-probe-then-render pattern.
One generic helper in `lib/integrations/status.ts` — useful when the next
integration (say, Klaviyo Profiles or Stripe) lands.

**Pagination hooks.** Shopify, Leads, Drafts, and Prospects clients each
reinvent cursor-based pagination slightly differently. Low priority — not
painful today — but worth a note for the next feature that wants infinite
scroll.

## 5. Skills + subagents to build

Not all of these are worth building. The ones that matter are the ones that
would have saved hours on the last two weeks of work. In priority order:

### Subagent — `jsonb-route-scaffolder` (M)
**What it does.** Takes a type name (`Play`, `Draft`, `Sender`), a list of
editable fields with shape (`brief: string`, `stage: enum(...)`,
`strategy: Partial<PlayStrategy>`), and outputs a `app/api/{name}/[id]/route.ts`
file implementing `PATCH` + `DELETE` with validation, shallow merge, and
activity logging. Output matches the existing pattern in
`app/api/plays/[id]/route.ts` exactly.

**Trigger.** "Scaffold PATCH + DELETE routes for drafts with fields
`subject`, `preheader`, `status`, `prospects`."

**Why.** The plays, drafts, suppressions, senders, and tasks routes are
variations on the same 150 lines. The next feature (keywords, landing
pages, etc.) will want the same boilerplate again. Build this once and new
routes go from thirty minutes to two.

### Subagent — `supabase-migration-scaffolder` (M)
**What it does.** Takes a table name + jsonb payload shape + RLS rules.
Emits the SQL for `apply_migration`, a TypeScript type in `lib/types.ts`,
and the `listX/getX/upsertX` helpers in `lib/dashboard/repository.ts`.

**Trigger.** "New table: `dashboard_landing_pages`, jsonb payload matches
`LandingPage` type, insert/update/delete for service role only."

**Why.** Every feature needs one. I've apply_migration'd something like a
dozen tables building the outreach pipeline and the shape of the call is
always the same.

### Skill — `play-strategy-scaffold` (S)
**What it does.** Given a one-paragraph brief, emits a full `PlayStrategy`
(hypothesis, sector, persona, messaging angles, success metrics,
disqualifiers) in Evari voice and at a sensible level of specificity.
Invokes `evari-copy` for tone.

**Trigger.** "Flesh out the strategy for: UK independent bike shops that
sell to SMEs."

**Why.** The Spitball chat inside a play does this today, but only after
the play exists. A standalone skill lets planning happen outside the app
— in a Slack DM, in a notebook — and drops a ready-to-paste JSON object.

### Reusable component library (not a skill) — `components/ui/editable/`
**What it does.** Exports `<EditableString>`, `<EditableTextarea>`,
`<EditableList>`, `<EditableNumber>`, `<EditableSelect>` with the exact
behaviour of the ones in `PlayDetailClient.tsx`: click to edit, Esc to
cancel, Enter to save (Cmd-Enter for multiline), optimistic state with
rollback on PATCH failure.

**Trigger.** Not an agent — just a refactor. Extract once and use
everywhere.

**Why.** PlayDetailClient has six of these now. Drafts detail will want
eight more. Senders detail, five more. Build the library today and the
next form takes ten minutes.

### Subagent — `play-dry-run-debugger` (M, optional)
**What it does.** Runs the dry-run pipeline step-by-step and reports
which targets were skipped and why (suppression match, no email, AI
failure, sender not set). Pipes the per-step diagnostics as a markdown
table.

**Trigger.** "Why did dry-run for play-xyz only generate 3 drafts?"

**Why.** Dry-run is the most complex chain in the app and when it
silently underperforms there's no good way to debug without re-reading
the whole file. Lower ROI than the scaffolders above, but high payoff
the next time it misbehaves. Build after the scaffolders ship.

### Things I would *not* build
- **`evari-signature-template` skill.** The signature template already
  lives in `lib/email/signature.tsx` with a `renderSignature` helper.
  Don't duplicate.
- **`test-send-email-generator` skill.** The "Send test" button does
  this in the UI. An agent would be slower than clicking.
- **`suppression-bulk-importer` subagent.** Niche — maybe runs twice a
  year. A one-line Python script when needed.
- **`inline-editor-ui-generator` subagent.** The component library in
  the section above solves this better than a code-generating agent.
  Agents are for one-offs; this is reusable code.

### Skill housekeeping
- **`evari-copy`** — add a section covering the outreach-email tone
  (short, question-led, no marketing jargon, compliant signoff) so
  drafts generated outside the dry-run route also hit the voice.
- **`evari-copy`** — add the hard rules from the signature template
  (no repeated logo, sender line format) so email bodies don't fight
  the footer.
- Document the jsonb + activity-log pattern in
  `lib/dashboard/repository.ts` as a header comment so future agents
  pattern-match from the source, not from tribal knowledge.
